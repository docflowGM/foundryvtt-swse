Vehicle Sheet v2 — End-to-End Design

A second-generation actor sheet for vehicles in the SWSE system, covering the sheet itself, the Shipyard authoring surface, the compendium importer, contextual crew resolution, the enhanced combat UI, and PiSWE-aware initiative integration. Written for the maintainer and future contributors — terse where the codebase already answers the question, longer where the design needs to land.

Repo foundryvtt-swse Surface sheets/v2/vehicle-sheet/ Targets Foundry v12+  ·  system v0.4.x

01Goals & Non-Goals

// what this redesign is, and explicitly is not

Goals

  • One sheet, all vehicles. Starfighter, capital, walker, speeder, station, mount — same actor type, conditional panels driven by system.category.
  • Editable when authoring, terse when playing. A read-mostly play view, with an "Edit" affordance that reveals the dense fields the GM cares about.
  • Crew is contextual. Pilot is the only required slot; other stations appear based on the vehicle's actual capabilities (weapons present → gunner, shields present → shield op, etc).
  • Houserule-aware. Enhanced Pilot/Engineer/Shields/Commander, SWES subsystems, vehicle turn controller, and the Ship-Based Initiative system all surface in the sheet only when their world settings are on.
  • Authorable. Drag from compendium or the world Item directory; import flows live in a single Shipyard router that all entry points delegate to.
  • v1 stays parseable. Existing world actors must round-trip without data loss; an idempotent migration pass adds v2 fields in place.

Non-Goals

  • Replacing VehicleRules or any engine module. The sheet consumes engine state; it does not own it.
  • Touching weapon-as-Item refactors — vehicle weapons stay in system.weapons[] for v2; a future v3 may promote them to Items.
  • Shipping a new combat tracker. PiSWE integration hooks into the existing combat-rules path, doesn't fork it.
  • Cross-system compatibility (foreign system imports). Out of scope for v1.
Design pillar

Every panel on the sheet has a single owner: either data (rendered from system), derived (rendered from VehicleRules / engines), or session (rendered from flags.swse.*). No panel mixes the three; no panel writes outside its own lane.
