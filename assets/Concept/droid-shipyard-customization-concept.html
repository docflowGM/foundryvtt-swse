Implementation Notes for Future V2 Build

  • This HTML is a concept only. No production engines, controllers, settings, or templates are introduced here.
  • The future production app should be an ApplicationV2 (Foundry v13) sheet/app.
  • UI should trigger engines only. Rule math, validation, slot counting, legality, costs, and any actor / item mutation must live in engines/ / services/, never in templates.
  • Droid Garage mode should consume the existing droid customization engines and data (chassis, locomotion, appendages, systems).
  • Starship Shipyard mode should consume the existing vehicle / starship customization engines and data (hull frames, upgrade slots, system bays, hardpoints).
  • Store mode should drive its prices, availability, vendor legality and black-market sourcing through the existing transaction / store engines.
  • Chargen droid mode should drive readiness, starting package completeness and unresolved-choice counts through the existing progression / chargen engines.
  • Tooltip contents (every data-swse-tooltip-key) should resolve through the existing SWSE tooltip registry / glossary — no inline strings, no title spam.
  • Theme tokens shown here mirror the existing --swse-theme-* / --swse-state-* vocabulary and must be supplied by the existing ThemeResolutionService + sheet theme settings + datapad shell tokens. No parallel theme system.
  • No parallel roll system. All rolls (validation rolls, install rolls if any) go through the existing roll service.
  • No direct actor / item mutation from UI controls. Buttons emit intents; engines apply mutation atomically.
  • Category color language (--cat-*) is intentionally distinct from semantic state colors and must not be reused for ability-score or signed-math UI.
  • Mode + context routing (Build New / Modify Existing / Store Quote / Chargen Draft) should be a single application state machine inside the unified shell — same shell, different lanes.
