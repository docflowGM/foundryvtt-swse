## Footer Navigation Architecture

### Template Structure

**Location:** `progression-shell.hbs` (lines 179-246)

**Three-Column Grid Layout:**
```html
<footer class="swse-footer" data-font-mode="...">
  <!-- LEFT: Back navigation -->
  <div class="action-footer__nav">
    <button data-action="previous-step" {{#unless footer.back.enabled}}disabled{{/unless}}>
      Back
    </button>
  </div>

  <!-- CENTER: Step progression chips -->
  <div class="action-footer__steps" role="tablist">
    {{#each visibleSteps}}
      <button data-action="jump-step" data-step-id="..."
              class="swse-step-btn {{state-classes}}">
        {{index}} {{label}}
      </button>
    {{/each}}
    {{!-- Status banners (blocking/warning) --}}
  </div>

  <!-- RIGHT: Next/Confirm -->
  <div class="action-footer__confirm">
    <button data-action="next-step|confirm-step"
            {{#unless footer.next.enabled}}disabled{{/unless}}>
      Next/Confirm
    </button>
  </div>
</footer>
```

## Current State Properties

### From Template Context

**`footer.back` (object)**
- `.enabled` (boolean) → Button disabled state
- `.label` (string) → Button text ("Back", etc.)

**`footer.next` (object)**
- `.enabled` (boolean) → Button disabled state
- `.isLastStep` (boolean) → Changes action & styling (confirm vs next)
- `.label` (string) → Button text ("Next" or "Confirm")

**`footer.blockingIssues` (array of strings)**
- Messages displayed in red banner
- Disables next button when present

**`footer.warnings` (array of strings)**
- Messages displayed in amber banner
- Warning state but allows progression

**`visibleSteps` (array of objects)**
Each step has:
- `.id` (string) → Step identifier
- `.index` (number) → Display number (1-based)
- `.label` (string) → Step name
- `.isCurrent` (boolean) → Current step (bright cyan)
- `.isComplete` (boolean) → Completed step (teal, navigable)
- `.isLocked` (boolean) → Future/locked step (grey, non-navigable)
- `.isWarning` (boolean) → Conditional/warning step (amber)
- `.canNavigate` (boolean) → Whether user can click to jump

## Progression Control Invariants (To Be Established)

### 1. Navigation Authority Rules

**Back Button (`previous-step`)**
```
Rule 1.1: Back should be disabled on first step
Rule 1.2: Back should jump to immediately previous step (step N-1)
Rule 1.3: Back can return to completed steps (user revises choices)
Rule 1.4: Back should NOT clear progress of skipped steps
```

**Next Button (`next-step`)**
```
Rule 2.1: Next should be disabled if blocking issues exist
Rule 2.2: Next should NOT be disabled for warnings (user can proceed)
Rule 2.3: Next should jump to step N+1
Rule 2.4: Next should verify current step is complete before allowing progression
Rule 2.5: On last step, Next becomes "Confirm" with `confirm-step` action
```

**Jump Button (`jump-step` + step-id)**
```
Rule 3.1: Only completed steps should be clickable (is-complete = true)
Rule 3.2: Current step should not be clickable (visual feedback)
Rule 3.3: Future locked steps should be disabled and non-clickable
Rule 3.4: Conditional warning steps should be clickable but visually distinct
Rule 3.5: Jumping backward should not clear progress of intermediate steps
Rule 3.6: Jumping forward should skip intermediate steps (if allowed)
Rule 3.7: Cannot jump forward past locked steps without unlocking
```

**Confirm Button (`confirm-step`)**
```
Rule 4.1: Only available on last step (footer.next.isLastStep = true)
Rule 4.2: Disabled if any blocking issues exist
Rule 4.3: Should finalize character progression
Rule 4.4: Triggers application completion flow
Rule 4.5: Cannot be undone (terminal action)
```

---

### 2. Step State Invariants

**Step Lifecycle:**
```
LOCKED → CURRENT → COMPLETE → [LOCKED again if revert]
```

**State Transition Rules:**
```
Rule 5.1: Step starts as LOCKED
Rule 5.2: Step becomes CURRENT when user navigates to it
Rule 5.3: Step becomes COMPLETE when user confirms/saves progress
Rule 5.4: CURRENT step cannot jump to another CURRENT step
Rule 5.5: COMPLETE step can return to CURRENT by navigating backward
Rule 5.6: Once COMPLETE, user can return to previous step N-1
Rule 5.7: Returning to previous step should preserve that step's progress
Rule 5.8: Future steps after current should remain LOCKED until reached
```

---

### 3. Blocking & Warning Invariants

**Blocking Issues (Red Banner)**
```
Rule 6.1: Blocking issues prevent progression (Next disabled)
Rule 6.2: Blocking issues are hard stops (e.g., missing required field)
Rule 6.3: Each blocking issue should be addressable in current step
Rule 6.4: Once resolved, Next button should re-enable automatically
Rule 6.5: Blocking issues clear when step is properly completed
Rule 6.6: Cannot jump forward while blocking issues exist
Rule 6.7: Can navigate backward even with blocking issues
```

**Warnings (Amber Banner)**
```
Rule 7.1: Warnings do NOT disable Next (user can proceed)
Rule 7.2: Warnings are informational (e.g., "This choice affects mechanics")
Rule 7.3: Multiple warnings can stack in banner
Rule 7.4: Warnings clear when underlying condition is resolved
Rule 7.5: Warning steps are marked as is-warning in chips
Rule 7.6: User can acknowledge and proceed past warnings
```

---

### 4. Visual State Consistency Rules

**Button Disabled Appearance:**
```
Rule 8.1: Disabled buttons show opacity: 0.38
Rule 8.2: Disabled buttons are non-interactive (cursor: not-allowed)
Rule 8.3: Disabled state persists visually without flickering
Rule 8.4: Disabled buttons should not respond to hover
```

**Step Chip States:**
```
Rule 9.1: CURRENT step: bright cyan, solid background
Rule 9.2: COMPLETE step: teal/green, clickable
Rule 9.3: LOCKED step: grey, dim, non-clickable
Rule 9.4: WARNING step: amber, distinct but navigable
Rule 9.5: Only one step can be CURRENT at a time
Rule 9.6: Step index badge always visible and readable
Rule 9.7: Step labels truncate with ellipsis if too long
```

**Last Step Special State:**
```
Rule 10.1: Last step's Confirm button has green teal glow
Rule 10.2: Last step's Confirm button pulses animation (2.4s cycle)
Rule 10.3: Animation stops on hover
Rule 10.4: Confirm button triggers completion (not reversible)
```

## Button State Matrix

| Button | Normal | Last Step | Back-Disabled | Next-Disabled |
|--------|--------|-----------|---------------|---------------|
| Back | enabled | enabled | **disabled** | enabled |
| Step Chips | varies by state | varies | enabled | enabled |
| Next/Confirm | enabled | enabled | enabled | **disabled** |

**Back Disabled When:**
- On first step (no previous to go to)

**Next Disabled When:**
- Blocking issues exist
- Step validation fails (step incomplete)
- On last step (hidden, replaced by Confirm)

**Step Chips Disabled When:**
- Step is LOCKED (future steps)
- Step is CURRENT (can't click own chip)

## Data Validation Requirements

### Before Allowing Previous Step:
```javascript
validate_previous() {
  // Rule 1.1: Cannot go back from first step
  if (currentStepIndex === 0) return false;

  // Rule 1.3: Back should work even with blocking issues (user revising)
  return true;
}
```

### Before Allowing Next Step:
```javascript
validate_next() {
  // Rule 2.1: Blocking issues prevent next
  if (footer.blockingIssues.length > 0) return false;

  // Rule 2.4: Current step must be complete
  if (!validateStepCompletion(currentStep)) return false;

  // Rule 2.3: Can't go past last step
  if (currentStepIndex >= totalSteps - 1) return false;

  return true;
}
```

### Before Allowing Jump:
```javascript
validate_jump(targetStepId) {
  // Rule 3.1: Target must be complete
  if (!isStepComplete(targetStepId)) return false;

  // Rule 3.2: Can't click current step
  if (currentStepId === targetStepId) return false;

  // Rule 3.7: Can't jump forward past locked steps
  if (isJumpingForward(targetStepId)) {
    for (let i = currentStepIndex + 1; i < targetStepIndex; i++) {
      if (isStepLocked(steps[i])) return false;
    }
  }

  return true;
}
```

### Before Allowing Confirm:
```javascript
validate_confirm() {
  // Rule 4.2: No blocking issues
  if (footer.blockingIssues.length > 0) return false;

  // Rule 4.1: Must be last step
  if (!isLastStep(currentStep)) return false;

  // Rule 4.3: Step must be complete
  if (!validateStepCompletion(currentStep)) return false;

  return true;
}
```

## Footer Rendering Rules

### When to Enable Back:
```
Enable ONLY if: currentStepIndex > 0
Disable if: currentStepIndex === 0
```

### When to Enable Next:
```
Enable ONLY if:
  - No blocking issues exist AND
  - Current step validation passes AND
  - Not on last step

Disable if:
  - Blocking issues exist OR
  - Current step incomplete OR
  - On last step
```

### When to Show/Enable Step Chips:
```
For each step:
  - Show: if step.index <= currentStepIndex + 1 (current + lookahead)
  - is-current: if step === currentStep
  - is-complete: if step.progress > 0 (user has visited)
  - is-locked: if step.index > currentStepIndex + 1
  - is-warning: if step.isConditional
  - Clickable (enabled): if step.isComplete AND step !== currentStep
  - Disabled: if step.isLocked OR step.isCurrent
```

### When to Show Status Banners:
```
Show blocking banner if:
  - footer.blockingIssues.length > 0
  - Display each issue on new line
  - Color: red (#EF4444)

Show warning banner if:
  - footer.warnings.length > 0 AND
  - No blocking banner shown (blocking takes precedence)
  - Display each warning on new line
  - Color: amber (#F59E0B)
```

### When to Show Confirm Button:
```
Change Next to Confirm if:
  - footer.next.isLastStep === true
  - Add class: swse-btn--last-step
  - Change action: next-step → confirm-step
  - Change label: "Next" → "Confirm"
  - Add pulse animation (2.4s)
```

## Accessibility Requirements

### Keyboard Navigation:
```
- Tab through buttons (Back, Step Chips in order, Next)
- Enter/Space to activate button
- Arrow keys to navigate step chips
- aria-selected on current step chip
- aria-label on chip container
```

### Focus Management:
```
- Back button gets focus after Previous action
- Next button gets focus after Next action
- Step chips are tabbable if not disabled
- Disabled buttons should be skipped by tab (or marked aria-disabled)
```

### Screen Reader Announcements:
```
- Step chip role="tab" with aria-selected
- Blocking/warning messages announced
- Disabled state announced
- Last step confirmed before final submission
```
