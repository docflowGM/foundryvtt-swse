                        RENDER QUEUE FIX — PATCH SUMMARY
═══════════════════════════════════════════════════════════════════════════════

ROOT CAUSE
──────────
The progression shell's render guard discarded nested rerender requests during
active renders. When Species (and other chargen) steps clicked an item, set
focusedItem, awaited mentor animation, then called render(), that rerender was
BLOCKED if the shell was still in an "isRendering" state.

Evidence: "[ProgressionShell] Render called while already rendering — BLOCKED"

Result: Detail rail never hydrated, left empty after mentor animation.

═══════════════════════════════════════════════════════════════════════════════

PERMANENT FIX — PROGRESSION-SHELL.JS
────────────────────────────────────

TWO CHANGES:

1. Constructor (line 287):
   Add one-line flag initialization:

   this._pendingRender = false;

2. render() method (lines 423–456):
   Replace hard-block with queue-on-demand logic:

   OLD (BROKEN):
   ─────────────────────────────────────────────
   if (this._isRendering) {
     console.warn("... BLOCKED ...");
     return this;
   }
   this._isRendering = true;
   // render work
   this._isRendering = false;
   return result;

   NEW (FIXED):
   ─────────────────────────────────────────────
   if (this._isRendering) {
     this._pendingRender = true;  // QUEUE instead
     return this;
   }
   this._isRendering = true;
   try {
     // render work
     return result;
   } finally {
     this._isRendering = false;
     if (this._pendingRender) {
       this._pendingRender = false;
       queueMicrotask(() => this.render());  // FLUSH
     }
   }

RISK ASSESSMENT
─────────────────

RISKS MITIGATED:
  ✓ Bounded Queue: Only queues ONE rerender, not unlimited
  ✓ Guard Release: try/finally ensures release even on error
  ✓ API Safety: Uses browser-standard queueMicrotask()
  ✓ Backward Compat: Existing code works unchanged
  ✓ Performance: Two renders per action is optimal

TESTING:
  ✓ Instrumentation covers all 17+ chargen steps
  ✓ Console logs prove queuing → flushing → hydration flow
  ✓ No infinite loops (queue is bounded to 1 item)
  ✓ No stack issues (uses microtask scheduling)
