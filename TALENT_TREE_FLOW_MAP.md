## ğŸ“‹ SUMMARY

This talent tree system follows a **multi-stage pipeline**:

1. **Initialization**: Load all talents from compendium into in-memory registry
2. **Normalization**: Organize by tree, normalize prerequisites
3. **Level Up/Chargen**: User selects class â†’ talent budget calculated
4. **Tree Selection**: Filter available trees by settings and class
5. **Visualization**: Build prerequisite graph, render SVG tree
6. **Validation**: Check prerequisites against actor state + pending selections
7. **Progression Engine**: Store talent IDs in `actor.system.progression.talents`
8. **Finalization**: Create Item documents from talent IDs, apply to actor
9. **Persistence**: Save progression state to flags, clear on completion

**Key Design Principles**:
- **Separation of Concerns**: Registry loads data, Validator checks rules, Engine manages state, UI renders
- **Deferred Item Creation**: Talents stored as string IDs until finalization
- **Atomic Updates**: All actor changes via `applyActorUpdateAtomic()`
- **State Recovery**: Progression state stored in flags for session continuity
- **Prerequisite Flexibility**: Supports both normalized and legacy prerequisite formats
