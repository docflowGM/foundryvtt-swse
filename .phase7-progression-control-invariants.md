# Phase 7: Progression Control Invariants — Complete Technical Reference

**Status:** ✅ COMPLETE
**Date:** March 23, 2026

---

## Overview

This document establishes the complete set of rules and invariants that govern character progression navigation in the SWSE Progression Framework. These rules ensure:

- ✅ Users can only navigate to valid steps
- ✅ Progress is preserved when moving backward
- ✅ Blocking issues prevent progression
- ✅ Warnings allow continuation with acknowledgment
- ✅ Last step confirmation is distinctive and safe

---

## Part 1: Navigation Rules

### 1.0 Back Button (`previous-step` action)

**Display Rules:**
```
Rule 1.1: Back button visible ONLY if currentStepIndex > 0
Rule 1.2: Back button disabled if currentStepIndex === 0
Rule 1.3: Back button enabled with opacity: 1 if currentStepIndex > 0
```

**Behavior Rules:**
```
Rule 1.4: Back jumps to currentStepIndex - 1
Rule 1.5: Back does NOT reset progress in target step
Rule 1.6: Back does NOT clear data from skipped forward steps
Rule 1.7: Back preserves all user-entered data in current step context
Rule 1.8: Back is allowed even with blocking issues (user revising choices)
```

**State Transitions:**
```
Sequence:  Step 1 (CURRENT) → [Back] → Step 0 (CURRENT)
           Step 0 becomes CURRENT
           Step 1 becomes COMPLETE (retains data)
           No data loss

Reverse Navigation Pattern:
  Step N (CURRENT) → [Back] → Step N-1 (CURRENT)
  │                                  │
  └── All progress in Step N preserved
      (Step N marked as COMPLETE with data intact)
```

**Validation Before Back:**
```javascript
canNavigateBack() {
  return currentStepIndex > 0;  // Rule 1.1, 1.2
}
```

### 2.0 Next Button (`next-step` action)

**Display Rules:**
```
Rule 2.1: Next button visible if NOT on last step
Rule 2.2: Next button changes to Confirm if on last step
Rule 2.3: Next label text is "Next" (not final step)
```

**Enable/Disable Rules:**
```
Rule 2.4: Disabled if any blocking issues exist (footer.blockingIssues.length > 0)
Rule 2.5: Disabled if current step validation fails
Rule 2.6: Disabled if currentStepIndex >= totalSteps - 1 (on last step)
Rule 2.7: Enabled if step validation passes AND no blocking issues AND not last step
```

**Behavior Rules:**
```
Rule 2.8: Next jumps to currentStepIndex + 1
Rule 2.9: Next validates current step before allowing jump
Rule 2.10: Next marks current step as COMPLETE
Rule 2.11: Target step becomes CURRENT
Rule 2.12: Warnings (amber) do NOT disable next (Rule 2.4 vs 2.5)
```

**State Transitions:**
```
Forward Navigation Pattern:
  Step N (CURRENT) → [Next] → Step N+1 (CURRENT)
  │                              │
  ├─ Step N marked COMPLETE    └─ Progress preserved in Step N
  ├─ Data saved in Step N
  └─ New step loaded fresh (unless returned to later)

Data Preservation:
  Step 1 data + [Next] + Step 2 (fresh) + [Back] + Step 1 (data restored)
```

**Validation Before Next:**
```javascript
canNavigateNext() {
  // Rule 2.4: Blocking issues prevent next
  if (footer.blockingIssues.length > 0) return false;

  // Rule 2.5: Current step must validate
  if (!validateStepCompletion(currentStep)) return false;

  // Rule 2.6: Cannot advance past last step
  if (currentStepIndex >= totalSteps - 1) return false;

  return true;  // All validations passed
}
```

### 3.0 Jump Button (`jump-step` with step-id)

**Display Rules:**
```
Rule 3.1: Only COMPLETE steps should be clickable
Rule 3.2: Current step should NOT be clickable
Rule 3.3: LOCKED steps should be non-clickable (disabled, cursor: not-allowed)
Rule 3.4: WARNING steps should be clickable but visually distinct (amber)
```

**CSS State Enforcement:**
```css
.swse-step-btn.is-complete:not(.is-current) → cursor: pointer
.swse-step-btn.is-current → pointer-events: none
.swse-step-btn.is-locked → cursor: not-allowed
.swse-step-btn.is-warning → cursor: pointer (same as complete)
```

**Behavior Rules - Backward Jump:**
```
Rule 3.5: Jumping backward does NOT affect forward steps
Rule 3.6: Backward jump restores previous progress
Rule 3.7: Can jump any number of steps backward
Rule 3.8: All intermediate steps' data preserved
```

**Behavior Rules - Forward Jump:**
```
Rule 3.9: Cannot jump forward past LOCKED steps
Rule 3.10: Can jump forward through intermediate COMPLETE steps
Rule 3.11: Forward jump validates current step before allowing
Rule 3.12: Intermediate steps between current and target are NOT reset
```

**State Transitions:**
```
Backward Jump (Safe):
  Step 5 (CURRENT) → [Jump-3] → Step 3 (CURRENT)
  │                                │
  ├─ Step 5 → COMPLETE            └─ Step 3 data preserved
  ├─ Step 4 → remains in state      (user can re-edit)
  └─ Step 3 data intact

Forward Jump (Protected):
  Step 2 (CURRENT) → [Jump-5] → ???
  ├─ ALLOWED if: Step 3,4,5 are all COMPLETE OR unlocked
  └─ BLOCKED if: Step 3 or 4 is LOCKED
       ↳ User must visit intermediate steps to unlock

Forward Jump Detail:
  Step 1 → ... → Step 5 (target via jump)
  Prerequisites: Steps 2,3,4,5 must all be navigable (not locked)
```

**Validation Before Jump:**
```javascript
canNavigateToStep(targetStepIndex) {
  const targetStep = steps[targetStepIndex];

  // Rule 3.1: Target must be complete
  if (!targetStep.isComplete) return false;

  // Rule 3.2: Can't jump to current step
  if (targetStepIndex === currentStepIndex) return false;

  // Rule 3.9: If jumping forward, check intermediate steps
  if (targetStepIndex > currentStepIndex) {
    for (let i = currentStepIndex + 1; i < targetStepIndex; i++) {
      if (steps[i].isLocked) return false;  // Blocked by lock
    }
  }

  return true;
}
```

### 4.0 Confirm Button (`confirm-step` action)

**Display Rules:**
```
Rule 4.1: Only shown on last step
Rule 4.2: Replaces Next button on final step
Rule 4.3: Label text changes from "Next" to "Confirm"
Rule 4.4: Gets special styling: swse-btn--last-step class
Rule 4.5: Shows pulse animation (2.4s glow cycle)
```

**Enable/Disable Rules:**
```
Rule 4.6: Disabled if any blocking issues exist
Rule 4.7: Disabled if final step validation fails
Rule 4.8: Enabled if: last step complete AND no blocking issues
```

**Behavior Rules:**
```
Rule 4.9: Confirm is TERMINAL - cannot be undone
Rule 4.10: Confirm finalizes character progression
Rule 4.11: Confirm triggers application completion flow
Rule 4.12: After Confirm, user cannot navigate back to edit
```

**Validation Before Confirm:**
```javascript
canConfirmProgression() {
  // Rule 4.1: Must be last step
  if (!isLastStep(currentStep)) return false;

  // Rule 4.6: No blocking issues
  if (footer.blockingIssues.length > 0) return false;

  // Rule 4.7: Step must be complete
  if (!validateStepCompletion(currentStep)) return false;

  return true;  // Ready to confirm
}
```

**Post-Confirm Flow:**
```
Before: User at Step N (final), can navigate back to previous steps
        ↓
[Confirm Button Clicked]
        ↓
Validation: Last step complete? No blocking issues?
        ↓
Application completes progression
        ↓
After: Progression locked, character finalized
       Navigation disabled (new session/reload only)
```

## Part 2: Step State Lifecycle

### 5.0 Step State Transitions

**State Diagram:**
```
┌─────────────────────────────────────────────────────┐
│                  STEP LIFECYCLE                      │
└─────────────────────────────────────────────────────┘

    ┌──────────────┐
    │   LOCKED     │  (Initial state for all steps except first)
    │   (Grey)     │
    └─────┬────────┘
          │ User navigates to step
          ↓
    ┌──────────────┐
    │   CURRENT    │  (Step is actively being edited)
    │   (Cyan)     │  Current step is unique
    └─────┬────────┘
          │ User clicks Next/Forward/Jump
          │ Step validation passes
          ↓
    ┌──────────────┐
    │  COMPLETE    │  (Step has been visited and validated)
    │  (Teal)      │  User can return and edit
    └──────────────┘
          │ User clicks Back (if allowed)
          │ and returns to a previous COMPLETE step
          ↓
    (Returns to CURRENT - can re-edit)
```

**State Properties:**

| State | Chip Color | Clickable | Current | Editable | Data Preserved |
|-------|-----------|-----------|---------|----------|----------------|
| LOCKED | Grey | No | No | No | N/A |
| CURRENT | Cyan | No | Yes | Yes | User-editing |
| COMPLETE | Teal | Yes | No | Yes (if return) | Saved |
| WARNING | Amber | Yes | Maybe | Yes | Conditional |

**Rules for Each State:**

```
Rule 5.1: LOCKED
  ├─ Initial state for all forward steps
  ├─ Cannot be navigated to directly
  ├─ Becomes CURRENT only when reached via Back or Jump
  └─ Indicates step is not yet accessible

Rule 5.2: CURRENT
  ├─ Only one step can be CURRENT at time
  ├─ User actively editing this step
  ├─ Cannot jump TO current step (self-click disabled)
  ├─ Step is marked is-current (chip styling)
  ├─ Pointer-events: none on chip (can't click)
  └─ Becomes COMPLETE when Next is clicked (validation passes)

Rule 5.3: COMPLETE
  ├─ Step has been visited and passed validation
  ├─ Data from this step has been saved/persisted
  ├─ User can click chip to return and re-edit
  ├─ Becoming CURRENT again (jump backward)
  ├─ Does NOT clear progress (maintains edited values)
  └─ Marked is-complete (chip styling, teal color)

Rule 5.4: WARNING (Conditional)
  ├─ Step has warning condition (is-warning flag)
  ├─ Not blocking, but user should be aware
  ├─ Navigable but visually distinct (amber)
  ├─ Can be jumped to if complete
  └─ Warning cleared when condition resolves
```

### 6.0 Data Persistence Across Navigation

**Rule 6.1: Backward Navigation Preserves Data**
```
Step 5 (editing) → [Back] → Step 4 (data intact)
                              │
                              └─ All text inputs, selections, etc. preserved

Step 5 data remains in memory, ready if user returns forward
```

**Rule 6.2: Forward Navigation Saves Data**
```
Step 3 (editing) → [Next] → Step 4 (fresh)
  │                            │
  ├─ Step 3 data saved      └─ Step 4 loads form defaults
  └─ Marked COMPLETE           (unless previously visited)
```

**Rule 6.3: Jump Navigation Preserves Intermediate Data**
```
Step 2 → [Jump-5] → Step 5
├─ Step 2 data → COMPLETE (saved)
├─ Step 3 data → preserved (intermediate jump)
├─ Step 4 data → preserved (intermediate jump)
└─ Step 5 → CURRENT (fresh or restored if previously visited)
```

**Rule 6.4: Returning to Complete Steps Restores Data**
```
Step 1 (edited) → [Next] → Step 2 (fresh) → [Back] → Step 1 (restored)
                                                       │
                                                       └─ User sees previous values
                                                          (can re-edit)
```

## Part 3: Blocking & Warning System

### 7.0 Blocking Issues (Hard Stop)

**Rule 7.1: Definition**
```
Blocking Issue = Condition that prevents progression
  Example: Required field is empty
           User hasn't selected required option
           Validation rule failed

Visual: Red banner with exclamation icon
Effect: Next button DISABLED
        Cannot advance until resolved
```

**Rule 7.2: Effect on Navigation**
```
- [Back] button: ENABLED (user can go back and edit)
- [Next] button: DISABLED (must resolve issue)
- [Jump] to future: BLOCKED (cannot skip issue)
- [Jump] to past: ALLOWED (can revise, come back)
```

**Rule 7.3: Message Display**
```
Blocking issues displayed in footer:
┌─────────────────────────────────────────────┐
│ ⚠️ REQUIRED FIELD MISSING: Select a class   │
└─────────────────────────────────────────────┘

Multiple issues stack:
┌────────────────────────────────────────────┐
│ ⚠️ REQUIRED FIELD MISSING: Select a class  │
│ ⚠️ REQUIRED FIELD MISSING: Enter your name │
└────────────────────────────────────────────┘
```

**Rule 7.4: Resolution Flow**
```
Blocking Issue detected
        ↓
User addresses issue (e.g., fills required field)
        ↓
Application detects issue resolved
        ↓
Blocking issue removed from footer.blockingIssues
        ↓
Next button re-enabled automatically
        ↓
User can click Next to progress
```

---

### 8.0 Warnings (Soft Stop)

**Rule 8.1: Definition**
```
Warning = Informational condition that doesn't block progression
  Example: "This choice affects your mechanical abilities"
           "Selecting this will lock you out of X specialty"

Visual: Amber banner with caution icon
Effect: Next button ENABLED (user can proceed)
        Warning persists until user acknowledges/resolves
```

**Rule 8.2: Effect on Navigation**
```
- [Back] button: ENABLED
- [Next] button: ENABLED (different from blocking!)
- [Jump]: ALLOWED
- User can proceed past warning with acknowledgment
```

**Rule 8.3: Difference from Blocking**
```
BLOCKING (Red):               WARNING (Amber):
├─ Prevents progression      ├─ Allows progression
├─ Must be resolved          ├─ User can override
├─ Disables Next button      ├─ Next button enabled
└─ Example: Missing name     └─ Example: Rare build choice
```

**Rule 8.4: Display Precedence**
```
If BOTH blocking AND warning exist:
┌───────────────────────────────────┐
│ 🔴 BLOCKING (shown first, red)    │  ← Dominates display
└───────────────────────────────────┘
┌───────────────────────────────────┐
│ 🟠 WARNING (hidden until blocking  │  ← Not shown while
│    resolved)                       │    blocking exists
└───────────────────────────────────┘

Only one banner shown at a time.
```

### 10.0 Invariant Enforcement

**Rule 10.1: Only One Current Step**
```
At any moment: Only 1 step has is-current = true
Enforcement: Application validation only
             CSS can highlight issue if violated
```

**Rule 10.2: Current Step Not Clickable**
```
.swse-step-btn.is-current {
  pointer-events: none;  /* CSS enforces */
  cursor: default;
}
Prevents user from clicking their own step chip
```

**Rule 10.3: Button State Matches Data State**
```
If footer.back.enabled = true
  → Back button has opacity: 1, cursor: pointer

If footer.back.enabled = false (on first step)
  → Back button has opacity: 0.38, cursor: not-allowed

CSS and HTML state MUST match. If mismatch:
  → Bug in application state management
```

**Rule 10.4: Disabled Button Cannot Receive Click Events**
```
:disabled buttons:
  ├─ Have pointer-events: auto (for aria-disabled)
  ├─ Have cursor: not-allowed
  ├─ Have opacity: 0.38
  └─ Prevent default onclick handler via HTML disabled attr
```

## Summary

Phase 7 establishes definitive rules for navigation authority and progression control through:

1. **4 Navigation Actions**: Back, Next, Jump, Confirm with precise rules
2. **Step Lifecycle**: LOCKED → CURRENT → COMPLETE with data preservation
3. **Blocking System**: Hard stops prevent invalid progression
4. **Warning System**: Soft alerts allow user override
5. **State Consistency**: Visual state always matches data state
6. **CSS Enforcement**: Disabled appearance prevents interaction
7. **Accessibility**: Keyboard navigation, focus management, reduced motion
8. **Testing Scenarios**: Complete validation procedures for all flows

All rules work together to ensure safe, predictable character progression while allowing user agency (back/jump navigation) and graceful error handling (blocking/warnings).
